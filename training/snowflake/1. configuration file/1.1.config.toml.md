# Snowflake CLI — `config.toml`

The `config.toml` file is the central configuration file for [Snowflake CLI](https://docs.snowflake.com/en/developer-guide/snowflake-cli/index) (`snow`). It lives in the **root of the project** and defines one or more **named connections** — each representing a Snowflake identity with its own user, role, warehouse, and authentication method.

---

## Why config.toml Matters for This Training

In a real data platform, **service accounts** (e.g. the dbt runner, the ingest pipeline) authenticate to Snowflake programmatically — without a browser or password prompt. They use dedicated users, roles, and warehouses scoped to exactly what they need.

Rather than deploying actual CI/CD pipelines, this training **emulates service accounts locally** by defining multiple named connections in `config.toml`. Switching connections is equivalent to switching identity — the same way a CI job would authenticate as `DATA_INGEST_USER` versus a data engineer logging in interactively.

```
Your machine
│
├── connection: admin       → authenticates as VITU144 / ACCOUNTADMIN
├── connection: local       → authenticates as you / your default role
├── connection: ingestion   → authenticates as DATA_INGEST_USER / ROLE_DATA_INGEST_DEV
└── connection: dbt         → authenticates as DBT_USER / ROLE_DBT_USER_DEV
```

---

## File Structure

```toml
default_connection_name = "local"   # used when -c is not specified

[connections]

[connections.<name>]
account          = "<account_identifier>"
user             = "<username>"
role             = "<role>"
warehouse        = "<warehouse>"
database         = "<default_database>"   # optional
authenticator    = "<method>"
# + auth-specific fields (see below)
```

### `default_connection_name`

The connection used when you run a `snow` command without specifying `-c`. Set this to the connection you use most interactively — typically your personal user connection (`local`).

### `account`

Your Snowflake account identifier. Format: `<orgname>-<account_name>` (e.g. `myorg-xy12345`). Find it in Snowsight under *Admin → Accounts*, or from the URL: `https://<account>.snowflakecomputing.com`.

**Those parameters and initial structure can also be found on Snowflake through the UI, by: 1.Click on your name; 2. Select "Connect a tool to Snowflake"; 3. Go to the second tab "Config File".**

## Authentication Methods

Each connection specifies an `authenticator` field. The project uses three methods:

---

### 1. Programmatic Access Token (PAT)

```toml
authenticator  = "PROGRAMMATIC_ACCESS_TOKEN"
token_file_path = "pat.txt"
```

A PAT is a long-lived token generated in Snowsight for a specific user. It is stored in a plain text file referenced by `token_file_path` (relative to where you run the `snow` command, or an absolute path).

**When to use:** Interactive human users and service accounts where key-pair setup is too heavy. Used by the `local`, `admin`, and `ingestion` connections in this project.

**How to generate a PAT:**
1. In Snowsight, click your username (bottom-left) → *My profile*
2. Scroll to *Programmatic access tokens* → *Generate new token*
3. Copy the token and save it to the file referenced in `token_file_path`

> The PAT file should **never be committed to git**. Add `*.txt` or the specific filename to `.gitignore`.

---

### 2. Key-Pair Authentication (JWT)

```toml
authenticator    = "SNOWFLAKE_JWT"
private_key_path = "/absolute/path/to/rsa_key.p8"
```

Uses an RSA private key to sign a JWT. The corresponding public key must be registered on the Snowflake user. No token file to manage — the key pair acts as the credential.

**When to use:** Service accounts that need a more durable auth method than a PAT, or where token rotation is automated. Used by the `dbt` connection in this project.

**How to set up:**
```bash
# Generate the key pair
openssl genrsa 2048 | openssl pkcs8 -topk8 -inform PEM -out rsa_key.p8 -nocrypt
openssl rsa -in rsa_key.p8 -pubout -out rsa_key.pub

# Register the public key on the Snowflake user (run as ACCOUNTADMIN)
ALTER USER DBT_USER SET RSA_PUBLIC_KEY='<contents of rsa_key.pub without headers>';
```

> Keep `rsa_key.p8` out of git. Add it to `.config`.

---

### 3. Browser SSO / Username+Password (reference)

Not used in this project but worth knowing:

```toml
# Browser-based SSO (opens a browser window)
authenticator = "externalbrowser"

# Username + password (not recommended for automation)
authenticator = "SNOWFLAKE"
password      = "..."
```

## The Project's Connections

This project defines four connections, each representing a different identity:

```toml
default_connection_name = "local"
```

### `local` — Your personal user

```toml
[connections.local]
account          = ""
user             = ""
role             = ""
warehouse        = ""
database         = ""
authenticator    = "PROGRAMMATIC_ACCESS_TOKEN"
token_file_path  = "pat.txt"
```

Your day-to-day interactive connection. Fill in your own `user`, `role`, and `warehouse`. Generate a PAT in Snowsight and save it to `pat.txt`.

---

### `admin` — ACCOUNTADMIN for setup scripts

```toml
[connections.admin]
account          = ""
user             = "VITU144"
role             = "ACCOUNTADMIN"
warehouse        = "COMPUTE_WH"
database         = ""
authenticator    = "PROGRAMMATIC_ACCESS_TOKEN"
token_file_path  = "a_pat.txt"
```

Used to run the RBAC bootstrap scripts (`2.2` through `2.6`). Authenticates with ACCOUNTADMIN privileges. Generate a separate PAT for this and save it to `a_pat.txt`.

> Using a separate connection (and token file) for admin work avoids accidentally running destructive operations with elevated privileges in day-to-day use.

---

### `ingestion` — Emulates the ingest service account

```toml
[connections.ingestion]
account          = ""
user             = "DATA_INGEST_USER"
role             = "ROLE_DATA_INGEST_DEV"
warehouse        = "WH_DATA_INGEST_DEV"
database         = "RAW_DEV"
authenticator    = "PROGRAMMATIC_ACCESS_TOKEN"
token_file_path  = "i_pat.txt"
```

Authenticates as `DATA_INGEST_USER` with `ROLE_DATA_INGEST_DEV`.

In production this would be a CI/CD job. Here you run it locally with `-c ingestion` to experience the same access boundaries.

Generate a PAT for `DATA_INGEST_USER` in Snowsight (log in as that user, or generate via ACCOUNTADMIN) and save it to `i_pat.txt`. **This step will be done during the training after going through the RBAC module.**

---

### `dbt` — Emulates the dbt service account

```toml
[connections.dbt]
account          = ""
user             = "DBT_USER"
role             = "ROLE_DBT_USER_DEV"
warehouse        = "WH_DBT_USER_DEV"
database         = "TRANSFORMED_DEV"
authenticator    = "SNOWFLAKE_JWT"
private_key_path = "/Users/vitorduarte/snowflake_dbt/rsa_key.p8"
```

Authenticates as `DBT_USER` with `ROLE_DBT_USER_DEV` using key-pair auth. This is the identity dbt uses to create schemas and run models in `TRANSFORMED_DEV`.

Update `private_key_path` to the absolute path of your `rsa_key.p8` file and register the public key on `DBT_USER` as described in the JWT section above.**This step will be done during the training after going through the RBAC module.**

## Setup Checklist

| Step | Action |
|------|--------|
| 1 | Copy `env.sample` to `.env` and fill in your account identifier |
| 2 | Set `account` in all four connections |
| 3 | Generate a PAT for your personal user → save to `pat.txt` |
| 4 | Generate a PAT for your admin session → save to `a_pat.txt` |
| 5 | Generate an RSA key pair → register public key on `DBT_USER` → update `private_key_path` |
| 6 | Generate a PAT for `DATA_INGEST_USER` → save to `i_pat.txt` |
| 7 | Run `snow connection test -c local` to verify your personal connection |
| 8 | Run `snow connection test -c admin` to verify ACCOUNTADMIN access |

## Testing a Connection

```bash
# Test connectivity
snow connection test -c local
snow connection test -c admin

# List databases visible to the ingestion identity
snow sql -q "SHOW DATABASES;" -c ingestion

# List databases visible to the dbt identity
snow sql -q "SHOW DATABASES;" -c dbt
```

Notice that `ingestion` only sees `RAW_DEV` and `dbt` only sees `TRANSFORMED_DEV` — that is the RBAC model from Module 2 in action.

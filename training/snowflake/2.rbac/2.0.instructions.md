# Module 2: Snowflake RBAC

## What You Will Learn

- How Snowflake's role-based access control works

- How to wire privilege grants to build a least-privilege data platform

---

## The RBAC Model

Snowflake RBAC is built around **roles** — not users. You grant privileges to roles, then assign roles to users. This makes access management auditable and easy to change.

This module implements a **three-layer model**:

```
┌─────────────────────────────────────────────────────┐
│  Layer 1: Ownership Roles                           │
│  Own the database objects. Prevent orphaned objects │
│  when users leave.                                  │
└───────────────────────┬─────────────────────────────┘
                        │ owns
┌───────────────────────▼─────────────────────────────┐
│  Layer 2: Access Roles                              │
│  READ / WRITE / CREATE — scoped to one database.   │
│  Schema Creator roles — CREATE SCHEMA only.        │
└───────────────────────┬─────────────────────────────┘
                        │ granted to
┌───────────────────────▼─────────────────────────────┐
│  Layer 3: Functional Roles                          │
│  Represent a persona (engineer, ingest SA, dbt SA). │
│  Inherit the access roles they need.                │
└─────────────────────────────────────────────────────┘
```

### Environments

Everything is duplicated across **DEV** and **PROD**. Engineers have full read/write access in DEV but read-only access to PROD. Service accounts in PROD can only do what their job requires.

### Functional Roles Summary

| Role | Persona | DEV Access | PROD Access |
|------|---------|------------|-------------|
| `ROLE_DATA_ENGINEER_{ENV}` | Human data engineer | RAW read+write, TRANSFORMED read+write | RAW read-only, TRANSFORMED read-only |
| `ROLE_DATA_INGEST_{ENV}` | Ingest service account | RAW read+write+create, schema create | RAW read+write+create, schema create |
| `ROLE_DBT_USER_{ENV}` | dbt service account | RAW read, TRANSFORMED read+write+create, schema create | RAW read, TRANSFORMED read+write+create |

## Files in This Module

Run them in order using Snowsight or snowcli (`snow sql -f <file> -c <connection>`).

---

### `2.1.manual.sql` — Authentication Policy

**Role required:** `ACCOUNTADMIN`

Creates a dedicated `ADMIN_DB` database with a `NETWORK` schema to store account-level policies. Defines an authentication policy called `TRAINING_SERVICE_ACCOUNT` that allows Personal Access Tokens (PATs) to bypass network policy evaluation.

**What to do:**
1. Replace `<YOUR USERNAME>` with your own Snowflake username and run the `ALTER USER` line to apply the policy to your account.
2. The two commented-out `ALTER USER` lines for `DBT_USER` and `DATA_INGEST_USER` will be applied in step 2.4 after those users are created.

> PATs are used instead of passwords for the service account users so you can authenticate programmatically without storing credentials.

---

### `2.2.rbac_roles.sql` — Create All Roles

**Role required:** `ACCOUNTADMIN`

Creates all roles in the hierarchy:

- **4 ownership roles** — one per database (`RAW_DEV`, `RAW_PROD`, `TRANSFORMED_DEV`, `TRANSFORMED_PROD`)
- **4 schema creator roles** — one per database, `CREATE SCHEMA` privilege only
- **12 access roles** — `READ`, `WRITE`, and `CREATE` for each of the 4 databases
- **8 functional roles** — `DATA_ENGINEER`, `DATA_INGEST`, `DBT_USER`, `DBT_SERVICE_ACCOUNT` for both DEV and PROD
- **1 aggregation role** — `ROLE_DATA_ENGINEER_ALL_READ` for cross-environment read access

All roles use `CREATE ROLE IF NOT EXISTS` so this script is safe to re-run.

---

### `2.3.rbac_dbs.sql` — Create Databases

**Role required:** `ACCOUNTADMIN`

Creates the four databases and immediately transfers ownership to the corresponding ownership role:

```
RAW_DEV          → ROLE_OWNER_RAW_DEV
RAW_PROD         → ROLE_OWNER_RAW_PROD
TRANSFORMED_DEV  → ROLE_OWNER_TRANSFORMED_DEV
TRANSFORMED_PROD → ROLE_OWNER_TRANSFORMED_PROD
```

**Why ownership roles?** If a human user creates a database, all objects belong to that user. When they leave, objects become orphaned. Ownership roles decouple object ownership from individual users.

---

### `2.4.rbac_users.sql` — Create Service Account Users

**Role required:** `USERADMIN`

Creates two service account users and assigns their functional roles:

| User | Default Role | Default Warehouse |
|------|-------------|-------------------|
| `DBT_USER` | `ROLE_DBT_USER_DEV` | `WH_DBT_USER_DEV` |
| `DATA_INGEST_USER` | `ROLE_DATA_INGEST_DEV` | `WH_DATA_INGEST_DEV` |

Both users are created with `MUST_CHANGE_PASSWORD = FALSE` since they will authenticate using PATs (configured in `2.1`).

**After running this file**, go back to `2.1.manual.sql` and run the two commented-out `ALTER USER` lines to apply the authentication policy to these users.

---

### `2.5.rbac_whs.sql` — Create Warehouses

**Role required:** `ACCOUNTADMIN`

Creates one warehouse per functional role per environment (8 total):

| Warehouse | Size | Used by |
|-----------|------|---------|
| `WH_DATA_ENGINEER_DEV` | XSMALL | `ROLE_DATA_ENGINEER_DEV` |
| `WH_DATA_INGEST_DEV` | XSMALL | `ROLE_DATA_INGEST_DEV` |
| `WH_DBT_USER_DEV` | XSMALL | `ROLE_DBT_USER_DEV` |
| `WH_DBT_SVC_DEV` | SMALL | `ROLE_DBT_SERVICE_ACCOUNT_DEV` |
| *(same pattern for PROD)* | | |

All warehouses are configured with `AUTO_SUSPEND = 60` and `INITIALLY_SUSPENDED = TRUE` to minimise cost.

**Why separate warehouses per role?** It enables precise cost attribution per persona and independent scaling — the ingest warehouse can be resized without affecting the engineer's warehouse.

### `2.6.rbac_grants.sql` — Privilege Grants ⚡ Challenge

**Role required:** `ACCOUNTADMIN`

This is the most important file. It wires everything together by granting privileges to roles. It is divided into five sections:

1. **Warehouse usage** — functional roles receive `USAGE` and `OPERATE` on their warehouse
2. **Schema creation** — schema creator roles receive `CREATE SCHEMA` on their database
3. **RAW database grants** — READ / WRITE / CREATE access roles for both DEV and PROD
4. **TRANSFORMED database grants** — **intentionally left blank for the challenge** (see below)
5. **Role hierarchy** — access roles are granted to functional roles; all roles granted to `SYSADMIN`

#### Challenge

Section 4 of `2.6.rbac_grants.sql` is empty:

```sql
-- ----- DEV TRANSFORMED -----
-- <TO BE FILLED IN THE TRAINING>
```

**Your task:** Write the `GRANT` statements for the `TRANSFORMED_DEV` database following the same pattern used for `RAW_DEV` in section 3. You need to cover three access roles:

- `ROLE_TRANSFORMED_DEV_READ` — database usage + SELECT on all/future tables and views
- `ROLE_TRANSFORMED_DEV_WRITE` — database usage + INSERT/UPDATE/DELETE/TRUNCATE on all/future tables
- `ROLE_TRANSFORMED_DEV_CREATE` — database usage + full object creation privileges on all/future schemas

**Hints:**
- The `TRANSFORMED` database needs `CREATE SCHEMA` in the CREATE role (unlike RAW, dbt creates its own schemas)
- Look at what `ROLE_RAW_DEV_CREATE` grants and consider which object types make sense for a transformation layer (`CREATE FUNCTION`, `CREATE PROCEDURE`, `CREATE DCM PROJECT`, `CREATE DBT PROJECT`)
- Check the diagram to see what the target state should look like

**Answer:** `2.8.CHECK.sql` contains the solution. Try to write it yourself first.

## Verification

After running all files, verify your setup in Snowsight:

1. **Roles** — navigate to *Admin → Users & Roles → Roles* and confirm all roles are present
2. **Users** — navigate to *Admin → Users & Roles → Users* and confirm `DBT_USER` and `DATA_INGEST_USER` exist
3. **Warehouses** — navigate to *Admin → Warehouses* and confirm all 8 warehouses are present
4. **Grants** — use snowcli to inspect what a role can see:

```bash
# What databases can the ingest user see?
snow sql -q "SHOW DATABASES" -c data_ingest

# Can the data engineer write to RAW_DEV?
snow sql -q "USE ROLE ROLE_DATA_ENGINEER_DEV; INSERT INTO RAW_DEV.SOURCE1.TEST VALUES (1);" -c data_engineer

# Can the data engineer create tables in RAW_DEV? (should fail)
snow sql -q "USE ROLE ROLE_DATA_ENGINEER_DEV; CREATE TABLE RAW_DEV.SOURCE1.TEST (id INT);" -c data_engineer
```
